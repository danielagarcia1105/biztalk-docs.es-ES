 <span data-ttu-id="2cee8-114">El inicio de sesión único (SSO) empresarial almacena en una base de datos de SSO una asignación cifrada entre el Id. de usuario de Windows de un usuario y sus credenciales en una o varias aplicaciones afiliadas.</span><span class="sxs-lookup"><span data-stu-id="2cee8-114">Enterprise Single Sign-On stores an encrypted mapping between a user’s Windows user ID and his credentials for one or more affiliate applications in an SSO database.</span></span> <span data-ttu-id="2cee8-115">Cuando este usuario necesita obtener acceso a una aplicación afiliada, puede buscar las credenciales de esa aplicación en la base de datos de SSO mediante un Servidor de inicio de sesión único (SSO).</span><span class="sxs-lookup"><span data-stu-id="2cee8-115">When this user needs to access an affiliate application, the credentials for that application can be looked up in the SSO database by a Single Sign-On (SSO) Server.</span></span> <span data-ttu-id="2cee8-116">El diagrama siguiente refleja su funcionamiento.</span><span class="sxs-lookup"><span data-stu-id="2cee8-116">The diagram below shows how this works.</span></span>

 <span data-ttu-id="2cee8-117">En este ejemplo, una orquestación procesa un mensaje enviado por alguna aplicación a BizTalk Server y lo envía a una aplicación afiliada que se ejecute en un gran sistema (mainframe) IBM.</span><span class="sxs-lookup"><span data-stu-id="2cee8-117">In this example, a message sent by some application to BizTalk Server is processed by an orchestration, then sent to an affiliate application running on an IBM mainframe.</span></span> <span data-ttu-id="2cee8-118">La misión del inicio de sesión único empresarial es garantizar que las credenciales correctas, es decir, el nombre de usuario y la contraseña correspondientes, se envían con el mensaje a la aplicación afiliada.</span><span class="sxs-lookup"><span data-stu-id="2cee8-118">The job of Enterprise Single Sign-On is to make sure that the correct credentials (that is., the right username and password) are sent with the message when it is passed to the affiliate application.</span></span>

 <span data-ttu-id="2cee8-120">Como muestra el diagrama, cuando un adaptador de recepción obtiene un mensaje, el adaptador puede solicitar un vale de SSO del servidor de SSO A (paso 1).</span><span class="sxs-lookup"><span data-stu-id="2cee8-120">As the diagram shows, when a receive adapter gets a message, the adapter can request an SSO ticket from SSO server A (step 1).</span></span> <span data-ttu-id="2cee8-121">Este vale cifrado contiene la identidad de Windows del usuario que realizó la solicitud y un periodo de tiempo en espera.</span><span class="sxs-lookup"><span data-stu-id="2cee8-121">This encrypted ticket contains the Windows identity of the user that made the request and a timeout period.</span></span> <span data-ttu-id="2cee8-122">(No se debe confundir con un vale de Kerberos, no es lo mismo.) Después de adquirirlo, se agrega como propiedad al mensaje entrante.</span><span class="sxs-lookup"><span data-stu-id="2cee8-122">(Don’t confuse this with a Kerberos ticket—it is not the same thing.)
