## <a name="a-contrl-receipt-may-report-a-status-of-accepted-when-part-of-the-message-is-rejected"></a><span data-ttu-id="8991b-107">Una recepción CONTRL puede informar de un estado aceptado cuando parte del mensaje se ha rechazado.</span><span class="sxs-lookup"><span data-stu-id="8991b-107">A CONTRL Receipt May Report a Status of Accepted when Part of the Message Is Rejected</span></span>  
 <span data-ttu-id="8991b-108">Una recepción CONTRL (confirmación técnica de EDIFACT) informa de un estado “Rechazado” sólo cuando el mensaje EDIFACT entrante es un duplicado o hay errores en el sobre (por ejemplo, existe un problema con el juego de caracteres).</span><span class="sxs-lookup"><span data-stu-id="8991b-108">A CONTRL receipt (EDIFACT technical acknowledgment) reports a status of “Rejected” only when the incoming EDIFACT message is a duplicate or there are errors in the envelope (for example, an issue with the character set).</span></span> <span data-ttu-id="8991b-109">EDIFACT no informa de un estado de intercambio “Aceptado con errores” en la confirmación técnica CONTRL, como hace X12 en el campo TA104 de una confirmación TA1.</span><span class="sxs-lookup"><span data-stu-id="8991b-109">EDIFACT does not report a state of “Interchange accepted with errors” in the CONTRL technical acknowledgment, as X12 does in the TA104 field in a TA1 acknowledgment.</span></span> <span data-ttu-id="8991b-110">Si se acepta parte del mensaje EDIFACT, la confirmación técnica CONTRL informará de un estado “Aceptado”.</span><span class="sxs-lookup"><span data-stu-id="8991b-110">If part of the EDIFACT message is accepted, the CONTRL technical acknowledgment will report “Accepted”.</span></span> <span data-ttu-id="8991b-111">En algunos escenarios, se rechazará parte del mensaje, pero la confirmación CONTRL seguirá informando de un estado “Aceptado”.</span><span class="sxs-lookup"><span data-stu-id="8991b-111">In some scenarios, part of the message will be rejected, but the CONTRL acknowledgment will still report a status of “Accepted”.</span></span> <span data-ttu-id="8991b-112">En tales escenarios, el elemento UCI5 puede informar del error.</span><span class="sxs-lookup"><span data-stu-id="8991b-112">In such scenarios, the UCI5 element may report the error.</span></span>

 <span data-ttu-id="8991b-114">Si la entrada procesamiento por lotes de opción para un mensaje está establecido en "Conservar intercambio: suspender intercambio en caso de error" y un campo en el encabezado de grupo o finalizador de X12 no es válida, se notificará el estado como aceptado en TA1 y 997 confirmaciones.</span><span class="sxs-lookup"><span data-stu-id="8991b-114">If the inbound batch processing option for an X12 message is set to “Preserve Interchange – suspend interchange on error”, and a field in the group header or trailer is invalid, the status will be reported as Accepted in both TA1 and 997 acknowledgments.</span></span> <span data-ttu-id="8991b-115">El informe de estado de EDI y los detalles del conjunto de transacciones también mostrarán un estado Aceptado.</span><span class="sxs-lookup"><span data-stu-id="8991b-115">The EDI status report and transaction set details will also indicate a status of Accepted.</span></span> <span data-ttu-id="8991b-116">Esto ocurre aunque se suspenda el intercambio y un error en el Visor de eventos indicará que se ha suspendido el intercambio.</span><span class="sxs-lookup"><span data-stu-id="8991b-116">This occurs even though the interchange will be suspended, and an error in the Event Viewer will indicate that the interchange was suspended.</span></span>  
  
 <span data-ttu-id="8991b-117">La confirmación TA1 mostrará un estado Aceptado porque está destinada a verificar la corrección del encabezado ISA y el finalizador IEA, pero no la corrección del encabezado GS y el finalizador GE.</span><span class="sxs-lookup"><span data-stu-id="8991b-117">The TA1 acknowledgment will show a status of Accepted because it is intended to verify the correctness of the ISA header and IEA trailer, but not the correctness of the GS header and GE trailer.</span></span> <span data-ttu-id="8991b-118">No obstante, la confirmación 997 también mostrará un estado Aceptado.</span><span class="sxs-lookup"><span data-stu-id="8991b-118">However, the 997 acknowledgment will also show a status of Accepted.</span></span>  
  
## <a name="if-groups-in-an-interchange-have-the-same-name-the-status-report-will-show-twice-as-many-acknowledgments"></a><span data-ttu-id="8991b-119">Si los grupos de un intercambio tienen el mismo nombre, el informe de estado mostrará dos veces tantas confirmaciones como haya.</span><span class="sxs-lookup"><span data-stu-id="8991b-119">If groups in an interchange have the same name, the status report will show twice as many acknowledgments</span></span>  
 <span data-ttu-id="8991b-120">Si el servidor BizTalk Server procesa un intercambio EDI con varios grupos que tienen el mismo nombre, el informe Intercambio EDI y estado de confirmación correlacionado enumerará dos veces tantas confirmaciones funcionales como se prevean.</span><span class="sxs-lookup"><span data-stu-id="8991b-120">If BizTalk Server processes an EDI interchange with multiple groups that have the same name, the EDI Interchange and Correlated ACK status report will list twice as many functional acknowledgments as expected.</span></span> <span data-ttu-id="8991b-121">Por ejemplo, si dos grupos de un intercambio tienen el mismo nombre, el informe de estado enumerará cuatro confirmaciones, en lugar de dos.</span><span class="sxs-lookup"><span data-stu-id="8991b-121">For example, if two groups in an interchange have the same name, the status report will list four acknowledgments, rather than two.</span></span>
