# <a name="key-monitoring-scenarios"></a>Escenarios de supervisión clave
El [!INCLUDE[btsBizTalkServerNoVersion](../includes/btsbiztalkservernoversion-md.md)] Management Pack para Operations Manager 2007 R2/2012 incluye una serie de escenarios de supervisión clave que se describe en la tabla siguiente.  
  
### <a name="biztalk-server-monitoring-scenarios"></a>Escenarios de supervisión de BizTalk Server  
  
|Escenario|Description|  
|--------------|-----------------|  
|Alertas de mensaje suspendido|Supervisar y resolver problemas relacionados con mensajes suspendidos son tareas operativas comunes en un entorno de BizTalk Server. El módulo de administración de BizTalk Server le permite supervisar y solucionar problemas de eventos de mensaje suspendido de una forma simplificada. Se producen dos tipos de suspensión de mensajes en BizTalk Server al procesar mensajes: suspensión de mensajes entrantes y salientes de suspensión de mensajes.<br /><br /> Para los mensajes entrantes, diversos tipos de procesamiento de errores o falta de suscripción pueden provocar un mensaje que se va a suspender. Algunos transportes pueden configurarse para suspender los mensajes o no suspenderlos. En lo que respecta a mensajes de salida, todos los errores de procesamiento conducirán invariablemente a la suspensión de mensajes.<br /><br /> Para los mensajes entrantes, las reglas de alerta de suspensión se escriben por adaptador. Este enfoque proporciona más flexibilidad en el control de suspensiones de mensajes en el lado de entrada. Puede agregar distintas acciones de respuesta en función del adaptador, como la notificación. Además, puede personalizar el módulo de administración de BizTalk Server mediante la creación de reglas personalizadas. Para los mensajes salientes, todos los mensajes se suspenden en caso de error de tiempo de transmisión de ningún tipo.|  
|Directiva de supresión de alertas|Reglas de mensajes suspendidos emplean una directiva de supresión distinta que generalmente se aplican para otras reglas. Las alertas se suprimen en función del nombre, origen, equipo o dominio que les corresponda.<br /><br /> Para la supresión basada en el origen de alerta, el origen de alerta para cada regla de mensaje suspendido también contiene un parámetro extraído del evento de mensaje suspendido. Este parámetro es el URI en el que se recibió el mensaje o al que se transmitió.<br /><br /> Errores que se producen en el mismo URI suelen ser debido a la misma razón. Por lo tanto, recomendamos que cree una sola alerta para la misma causa principal, aunque existen varios eventos de error asociados. Esto le permite tener una alerta y causa una raíz que resulta más fácil para que pueda corregir problemas cuando se produzcan. Alertas simplificadas, basadas en identificadores URI, reducen alertas innecesarias y facilitan un seguimiento de los problemas abiertos. Cuando se produce una suspensión de mensaje en un URI sin alertas de suspensión anteriores, se crea una nueva alerta.|  
|Alertas de asistencia técnica de BAM|Business Activity Monitoring (BAM) Portal los usuarios, conocidos como los usuarios empresariales, normalmente entienden los indicadores de nivel empresarial y actividades económicas. Usan el Portal de BAM para indicadores de agregados de monitor, las transacciones de negocio en curso o buscan transacciones con determinadas propiedades para investigar en profundidad. No obstante, los usuarios empresariales a menudo comprender los aspectos de un problema en un nivel de la infraestructura de TI. Durante la investigación de una transacción determinada, el Portal de BAM permite a los usuarios empresariales solicitar asistencia técnica desde el personal de operaciones de TI para investigar transacciones en el nivel de la infraestructura de TI.<br /><br /> El portal de BAM hace posible que se realice esta entrega del usuario empresarial mediante una solicitud de asistencia técnica. Al solicitar asistencia técnica, crea una entrada en el registro de aplicación en el Visor de eventos del equipo donde se hospeda el Portal de BAM. Este evento contiene detalles sobre la actividad en el que se solicita asistencia técnica por el usuario y otra información importante, como la información de grupo de identificador de la instancia de mensaje, el identificador de la instancia de servicio y BizTalk Server.<br /><br /> El módulo de administración de BizTalk Server contiene una regla para desencadenar una alerta en la detección de un evento de asistencia técnica de BAM. La regla que crea esta alerta es "Error: asistencia técnica de BAM necesario." Se crea una nueva alerta para cada solicitud enviada desde el Portal de BAM.|  
|Hosts y cuadros de mensaje de BizTalk|Módulo de administración de BizTalk Server incorpora reglas de umbral de rendimiento que proporcionan una vista completa del estado de BizTalk MessageBox bases de datos y las colas. Se proporcionan dos tipos de reglas de umbral:<br /><br /> -Reglas que se aplican de forma genérica, es decir, para todos los Hosts de BizTalk, todas las bases de datos de BizTalk MessageBox.<br />-Reglas específicas a un BizTalk Host determinado o un cuadro de mensajes.<br /><br /> Las reglas genéricas supervisan todos los Hosts de BizTalk Server o bases de datos de cuadro de mensajes según un umbral común. Por ejemplo, la regla “Monitor HostQ Size” supervisa las colas de trabajo de todos los hosts de BizTalk Server en virtud de un umbral común. Si existen tres hosts, todos sus colas de trabajo están supervisados por la misma regla y alertas se producen cuando cualquiera de las colas de trabajo de host cruza el umbral común.<br /><br /> Reglas de específico del host de BizTalk Server permiten configurar varios umbrales para distintos hosts y las bases de datos de cuadro de mensajes. Por ejemplo, la regla "Monitor HostQ Size – BizTalkServerApplication" es una regla de host específico que supervisa la cola de trabajo del host BizTalkServerApplication. Para ello, si define un proveedor concreto de Operations Manager para la instancia de contador de rendimiento determinado y utilizando ese proveedor en la regla de umbral.<br /><br /> Las reglas de host o de cuadro de mensajes específicas se proporcionan como reglas de plantilla para que se utilicen como una guía para la creación de reglas aplicables en su entorno. Las reglas genéricas deben configurarse con valores de umbral específicos de su entorno. Las reglas de host o de cuadro de mensajes específicos de BizTalk deben crearse de acuerdo con umbrales adecuados y reglas de plantilla.<br /><br /> BizTalk Server incorpora autolimitación que ayuda a evitar la sobrecarga se basa en varios parámetros. Una sobrecarga temporal que da lugar a una limitación no es un evento significativo operativamente hablando. Por el contrario, una limitación persistente constituye un evento inesperado en un entorno estable que podría indicar la existencia de problemas subyacentes en el nivel de la infraestructura. El módulo de administración de BizTalk Server proporciona supervisión proactiva de dichas condiciones de limitación persistentes con monitores de umbral de rendimiento en función de las condiciones siguientes:<br /><br /> -Memoria de proceso del servicio de BizTalk Server<br />-Número de mensajes procesados<br />-Número de subprocesos de un proceso de BizTalk Server<br />-El tamaño de las colas de base de datos de BizTalk|  
|Errores de orquestaciones|Supervisión de orquestaciones suspendidas es otro escenario común. Las orquestaciones pueden tener errores por muchas razones por las que se incluyen problemas del motor de orquestación, errores de componentes de .NET y la lógica del programa. Es importante habilitar un operador para determinar rápidamente el origen del problema y reanudar orquestaciones suspendidas cuando fija.|  
|Supervisión del servicio|BizTalk Server requiere que se está ejecutando un número de servicios para que funcione correctamente. El estado de estos servicios se asignarán a los monitores para las clases que representan los diferentes servicios de BizTalk.|  
|Supervisión de aplicaciones|BizTalk Server se introdujo el concepto de una aplicación para agrupar los artefactos de la solución. Supervisión de aplicaciones detecta los efectos de un error en el nivel de aplicación.|
